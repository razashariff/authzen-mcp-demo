AuthZEN MCP Demo v0.1 reference implementation

RFC 9421 Agent-Signature verification as an AuthZEN PDP context attribute, applied to MCP tool calls.

Reference implementation only. Not affiliated with the OpenID Foundation. Companion to the AuthZEN MCP Profile v0.1 discussion draft. The author has not yet executed the OpenID Foundation IPR Contribution Agreement.

What this demo shows

An AuthZEN PDP receives an authorisation request whose context contains an agent_signature. The PDP verifies the RFC 9421 canonical request locally, checks the action against a simple policy, and returns a standard AuthZEN decision. If anything in the request body has been modified after signing, the signature fails verification and the decision flips to Deny.

  • Signing key: ECDSA P-256, generated at server start, published as JWKS at /.well-known/jwks.json
  • Canonical request: method, target URI, ISO-8601 timestamp, SHA-256 body hash, agent identity claims
  • Verification: Node.js standard library only (crypto.verify) -- ~10 lines, no SDK, no third-party service
  • AuthZEN PDP API: POST /access/v1/evaluation with the AuthZEN v1 request shape

Try it

The flow

  1. Demo agent constructs an MCP tool call (a transfer_funds JSON-RPC body).
  2. Agent computes the canonical request and signs it with its ECDSA P-256 private key.
  3. If the "tamper" flag is set, the request body is modified after signing (amount becomes 999999).
  4. The MCP request is wrapped in an AuthZEN evaluation request with the signature in the context.
  5. The PDP recomputes the canonical request from the (possibly tampered) body and verifies the signature.
  6. The PDP applies its policy. The final decision is signature-valid AND policy-allows.
  7. The decision is returned in standard AuthZEN format.

Why this matters for the AuthZEN MCP Profile

The AuthZEN MCP Profile Section 1 currently defines how token claims become the AuthZEN subject. This demo shows the immediately adjacent question -- how the PEP establishes that the request body actually originated from the claimed subject and was not modified in flight -- can be answered with a profile of RFC 9421 (HTTP Message Signatures) as a context attribute. No new cryptography. No changes to the AuthZEN core API. Same shape as the Mastodon, ActivityPub, and payment-network profiles of RFC 9421.

Endpoints

POST /access/v1/evaluation
AuthZEN v1 PDP API. Accepts an evaluation request with optional context.agent_signature. Returns a standard AuthZEN decision.
POST /demo/tool-call
Simulates a full MCP tool call going through the PDP. Body: { "tamper": false }.
POST /demo/tamper-test
Runs the honest and tampered cases in one call.
GET /.well-known/jwks.json
The demo agent's public key.
GET /health
Service liveness probe.
